# 有了 Claude Code 源码，还需要 LangGraph 吗？

刚在自己项目做agent架构设计，一开始，我先让claude code自己调研了一波后自己写方案，
我看他做了各种方案优缺点对比后，基于我们的技术需求，最后推荐拿langGraph来改造，

我就接着追问了他几个问题，几个问题后，他改而说我们要不基于claude code来改造吧，
有点意思，
> 问claude code的几个问题： 
>1. 我们为什么基于langgraph，？基于我们的场景与需求，优缺点是什么?
>2. 我们真有必要用langGraph 的Checkpoint 吗？我们的需求不就是个数据库存json 的事吗？ 
>3. interrupt 我们为什么必须用？就为了这个而选langGraph？Claude Code(我早就把源码给他自己分析了) 怎么做的？


遂让claude code自己写了这篇文章，记录下我的分析过程，也许对同样在做 Agent 选型的同学有参考。

---

## 先说结论

LangGraph 主打三个东西：
1. **图式工作流**（StateGraph）——用有向图定义 Agent 流程
2. **Checkpoint**——自动把状态存到数据库
3. **interrupt()**——暂停等用户输入

看着都很好对吧？但仔细一想，每一个都有更简单的替代。

---

## LangGraph 卖点 1：图式工作流

### LangGraph 的思路

把 Agent 流程画成一张有向图——节点是"步骤"，边是"条件路由"：

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│ 搜索资料 │────▶│ 写文章   │────▶│ 审核     │
└──────────┘     └──────────┘     └────┬─────┘
                                      │    │
                              通过 ◀──┘    └──▶ 打回修改
                                │              │
                           ┌────▼─────┐        │
                           │ 完成输出 │  ┌─────▼─────┐
                           └──────────┘  │ 回到"写文章"│
                                         └───────────┘
```

看着很清晰，对吧？

### Claude Code 的做法

Claude Code 没有图。它就是一个循环：

```
┌──────────────────────────────────────────────────┐
│                                                    │
│           ┌───────────────┐                        │
│    ┌─────▶│  问 LLM 下一步 │                        │
│    │      └───────┬───────┘                        │
│    │              │                                │
│    │       LLM 要调工具？                            │
│    │        ╱        ╲                              │
│    │       是          否                            │
│    │       │           │                            │
│    │  ┌────▼────┐  ┌───▼───────┐                   │
│    │  │执行工具  │  │输出文本给   │                   │
│    │  │把结果给LLM│ │用户，结束   │                   │
│    │  └────┬────┘  └───────────┘                   │
│    │       │                                        │
│    └───────┘  ◀── 带着工具结果继续循环               │
│                                                    │
└──────────────────────────────────────────────────┘
```

Claude Code 用这一个循环驱动了 43 个工具——代码搜索、文件编辑、终端执行、多 Agent 协作，全靠它。

### 为什么图是多余的？

这个问题的关键在于——**谁在决定"下一步做什么"？**

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                   │
│  LangGraph 的图：                                                 │
│                                                                   │
│   代码预定义的边 ─── 决定 ──── 下一步去哪个节点                    │
│   （写死在代码里的 if/else）                                       │
│                                                                   │
│                                                                   │
│  Claude Code 的循环：                                              │
│                                                                   │
│   LLM 自己 ─── 决定 ──── 调什么工具 = 下一步做什么               │
│   （LLM 看了上下文，自己判断）                                     │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

**LLM 本身就是路由器。** 它根据对话上下文自己决定该搜索还是该写文件还是该问用户。图的"条件边"做的事情，LLM 的 tool_use 已经做了。

而且后者更灵活——LLM 可以根据情况随时改变策略，不受预定义的图的约束。

## LangGraph 卖点 2：Checkpoint

### LangGraph 的思路

"每个节点执行后，自动把整个状态存到 PostgreSQL。用户下次回来，从 Checkpoint 恢复。"

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│ 节点 A  │────▶│ 节点 B  │────▶│ 节点 C  │
└────┬────┘     └────┬────┘     └────┬────┘
     │               │               │
     📸              📸              📸
  存档 1          存档 2          存档 3
 (PostgreSQL)   (PostgreSQL)   (PostgreSQL)
```

### 想一想，这不就是"存对话历史"吗？

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                   │
│  LangGraph Checkpoint 存的：                                      │
│  { state: { messages, current_phase, todos, results, ... } }     │
│  ↓                                                                │
│  序列化整个 State 对象 → 存 PostgreSQL → 恢复时反序列化            │
│                                                                   │
│                                                                   │
│  自己存消息就行：                                                  │
│  [                                                                │
│    { role: "user", content: "帮我分析一下这段代码的性能问题" },    │
│    { role: "assistant", content: "好的，我先看看代码..." },       │
│    { role: "tool", content: "文件内容: function xxx()..." },      │
│    { role: "assistant", content: "找到了，第 42 行有个 N+1..." }, │
│  ]                                                                │
│  ↓                                                                │
│  JSON 存 PostgreSQL → 恢复时加载 JSON → LLM 看到历史继续          │
│                                                                   │
│  效果完全一样。LLM 不关心你是从 Checkpoint 恢复的                  │
│  还是从 JSON 加载的——它只看 messages。                             │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

**消息就是状态。** 不需要额外搞一层 Checkpoint。

## LangGraph 卖点 3：interrupt()

### LangGraph 的思路

"Agent 执行到一半需要用户确认？用 interrupt() 暂停，用户回来后恢复。"

```
执行节点A → 执行节点B → interrupt() 🛑 暂停！
                                      ⬇️ 等用户...
                                      ⬇️ 用户回来了
                                      恢复 → 执行节点C → 完成
```

### 但 Web 对话根本不需要这个

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                   │
│  用户: "帮我重构这个模块的数据库查询"                              │
│                                                                   │
│  Agent 处理（一次 HTTP 请求内）:                                   │
│    LLM → 读文件 → LLM → 分析代码 → LLM →                        │
│    "我找到了 3 处可以优化的地方，你看看方案？"                      │
│     ↑                                                             │
│     LLM 自己决定"该问用户了"                                      │
│     → 不调工具 → 循环自然结束 → HTTP 响应返回                     │
│                                                                   │
│  ════ 用户去吃饭了... 2 小时后回来 ════                            │
│                                                                   │
│  用户: "第 2 个方案改成用 JOIN 而不是子查询"                        │
│                                                                   │
│  Agent 处理（新的 HTTP 请求）:                                     │
│    从数据库加载历史消息 →                                          │
│    LLM 看到之前的 3 个方案 → 知道要改什么 → 修改 → 返回           │
│                                                                   │
│  完全不需要"暂停图执行"。                                          │
│  Web 对话天然就是"一问一答"，每个请求就是一个完整周期。            │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

**LLM 自己决定何时问用户**——比代码里硬写 interrupt() 更灵活。LLM 会根据修改的大小判断"这个改动大，我先确认一下"还是"小修改，直接做了"。

## Claude Code 比 LangGraph 多了什么？

Claude Code 的循环虽然简单，但围绕它有几个 LangGraph 没有的生产级设计：

### 1. 三层消息压缩

对话太长 token 爆了怎么办？LangGraph 没有内置方案。

Claude Code 从便宜到贵有三层策略：

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                   │
│  第 1 层: Snip（免费）                                            │
│  把早期的工具调用记录删掉                                          │
│  "反正 LLM 不需要知道 3 小时前搜了什么"                           │
│                                                                   │
│          ↓ 还不够？                                               │
│                                                                   │
│  第 2 层: Microcompact（免费）                                    │
│  工具返回了 5000 字的搜索结果 → 压缩成一句话摘要                  │
│  "搜索了 10 条结果，最相关的是 utils/auth.ts 的认证逻辑"          │
│                                                                   │
│          ↓ 还不够？                                               │
│                                                                   │
│  第 3 层: Autocompact（便宜，用小模型）                           │
│  用 Haiku 把整个对话做一次全文摘要                                │
│  "之前讨论了 3 处数据库优化，用户偏好 JOIN 而非子查询..."         │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

每一层都经过 Anthropic 的生产验证。对于长对话的 Agent 来说，这东西比 Checkpoint 重要多了。

### 2. 流式工具执行

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                   │
│  LangGraph 的顺序:                                                │
│  LLM 输出完整响应 ──▶ 解析工具 ──▶ 串行执行 ──▶ 下一个节点        │
│  ├── 3 秒 ──────────┤├── 1.5 秒 ────────────┤ = 4.5 秒           │
│                                                                   │
│  Claude Code 的方式:                                               │
│  LLM 边输出 ──▶ 工具 block 一完整就立刻执行（并行）               │
│  ├── 3 秒 ──────────────────────────┤                              │
│       └── 工具1/2/3 同时跑(0.5秒) ──┘ = 3.5 秒                   │
│                                                                   │
│  一轮省 1 秒，Agent 循环 5 轮 = 省 5 秒。用户体感差别很明显。     │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

### 3. 收益递减检测

LLM 有时候会空转——连着几轮调工具但没有实质进展。Claude Code 检测到这个模式后自动停下来，不白花钱。LangGraph 没有这个。

### 4. 依赖注入（可测试性）

Claude Code 的核心只依赖 4 个外部接口——调 LLM、压缩消息、存储、生成 ID。全部可以替换成 mock。写单元测试不需要真的调 Claude API，不需要连数据库。

## 那到底该选哪个？


**主要看你的agent到底谁来驱动** ？ 
而这个，底层又基于2个比较重要的因素
1. 你使用的llm，推理能力怎么样？ 你到底是用doubao还是opus、gemini？ 
2. 你对自己调教的system prompt，在你选用的llm模型上的表现，有信心不？ 
> 或者说，当你有一个手下时，他干活是自己决策还是需要你决策？而这又取决于他的能力(推理能力)和你给他的指导（system prompt）


```
┌──────────────────────────────────────────────────────────────────┐
│                                                                   │
│  你的 Agent 流程是谁在驱动？                                      │
│                                                                   │
│  ┌────────────────────┐        ┌──────────────────────┐          │
│  │ 代码驱动            │        │ LLM 驱动             │          │
│  │                    │        │                      │          │
│  │ 步骤固定            │        │ LLM 自己决定          │          │
│  │ 分支写死在代码里    │        │ 调什么工具、什么顺序  │          │
│  │                    │        │ 什么时候问用户        │          │
│  │ 例：ETL 管道       │        │                      │          │
│  │     审批流          │        │ 例：编程助手          │          │
│  │     数据处理        │        │     创作助手          │          │
│  │                    │        │     研究助手          │          │
│  │       ↓            │        │       ↓              │          │
│  │   LangGraph ✅     │        │  参考 Claude Code ✅  │          │
│  └────────────────────┘        └──────────────────────┘          │
│                                                                   │
│  不确定？                                                         │
│  → 先用 LangGraph 搭原型（快），跑通了再决定要不要自研             │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

### 更详细的对照

| 场景 | 推荐 | 理由 |
|------|------|------|
| ETL / 审批流 / 固定流水线 | LangGraph | 步骤固定，图让流程可视化 |
| 团队不太懂 Agent 原理 | LangGraph | 框架降低心智负担 |
| 快速验证 idea | LangGraph | 搭原型比自研快 |
| LLM 驱动的对话式 Agent | 参考 Claude Code | LLM 自己路由，不需要图 |
| 需要流式 + 消息压缩 | 参考 Claude Code | LangGraph 没有内置 |
| Web SaaS 多用户 | 参考 Claude Code | while 循环 + 数据库，天然无状态 |
| 对性能/成本敏感 | 参考 Claude Code | 去掉框架开销，直接控制 |
