## 一、总体对比概览

| 维度 | Claude Code | OpenClaw |
|------|------------|----------|
| **定位** | IDE 内嵌代码助手 CLI | 多渠道个人 AI 助手平台 |
| **核心场景** | 软件工程任务 | 全场景个人助手（代码+生活+消息+设备） |
| **提示词规模** | ~770 行（含工具描述） | ~676 行主提示词 + 多个辅助提示词 |
| **动态程度** | 静态核心 + 动态环境信息 | 高度动态（按工具/频道/模式/沙箱条件拼接） |
| **多智能体** | Coordinator + Worker 模式 | 多层嵌套子智能体（递归生成，深度控制） |
| **通信渠道** | 仅终端/IDE | 20+ 渠道（WhatsApp、Telegram、Slack、Discord、Signal、iMessage 等） |
| **提示词模式** | 单一模式（外部 vs 内部用户有轻微差异） | 三级模式（full / minimal / none） |
| **缓存策略** | 静态/动态分界线（`SYSTEM_PROMPT_DYNAMIC_BOUNDARY`） | 无显式缓存分界（但 minimal/none 模式起类似作用） |
| **安全模型** | 详细的操作安全指南 + 安全编码指导 | 简洁的 AI 安全宪法 + 外部硬执行（策略/沙箱） |

## 二、架构与设计理念对比

### Claude Code 的设计理念

**"软件工程专家"模式** — Claude Code 的提示词围绕一个核心目标：成为最优秀的编程搭档。

- **深度的编码规范**：详尽的 Git 安全协议、代码风格指导（不过度工程化、不添加多余注释等）、OWASP 安全编码
- **操作安全至上**：大量篇幅描述哪些操作需要确认、哪些可以自由执行、如何判断操作风险
- **工具优先于 Bash**：强烈要求使用专用工具而非通用 shell 命令
- **静态+动态分离**：通过 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 实现缓存优化

### OpenClaw 的设计理念

**"全能个人助手"模式** — OpenClaw 的提示词围绕多渠道、多模态、多智能体的个人助手体验。

- **渠道多样性**：支持 20+ 通信渠道，提示词需处理不同渠道的特性（回复标签、内联按钮、反应/表情等）
- **分级提示词**：通过 `promptMode` 为不同角色（主智能体 vs 子智能体 vs 最小实例）提供不同粒度的指令
- **人格化**：通过 SOUL.md 注入个性、语气和举止
- **推送而非轮询**：子智能体完成通知采用推送模式，减少无效轮询

## 三、功能模块逐项对比

### 3.1 身份定义

| 对比点 | Claude Code | OpenClaw |
|--------|------------|----------|
| 身份声明 | "You are Claude Code, Anthropic's official CLI for Claude." | "You are a personal assistant running inside OpenClaw." |
| 品牌归属 | 明确的 Anthropic 品牌绑定 | 中性平台描述，不绑定特定模型提供商 |
| 人格系统 | 无独立人格系统 | 通过 SOUL.md 注入完整人格（语气、举止、情绪） |
| 身份扩展 | 固定身份 | 可通过 IDENTITY.md、USER.md 动态扩展 |

**评价**：
- Claude Code 的品牌绑定使身份清晰但不灵活
- OpenClaw 的中性定位+人格注入系统更灵活，但也意味着没有人格文件时，助手缺乏个性

### 3.2 安全与防护

| 对比点 | Claude Code | OpenClaw |
|--------|------------|----------|
| 安全篇幅 | ~30 行安全相关指令 + 大量操作安全指导 | ~4 行安全宪法 + 外部硬执行 |
| 操作安全 | 详细的破坏性操作清单、逐条风险分类 | 无显式操作安全指导（依赖工具策略和沙箱） |
| 安全编码 | OWASP Top 10 警示、具体漏洞类型列举 | 无安全编码指导 |
| 网络安全 | 专门的 `cyberRiskInstruction`（授权测试、CTF、双用途工具） | 无网络安全指令 |
| AI 安全 | 隐式（不直接提及 AI 安全宪法） | 显式引用 Anthropic 宪法；明确禁止自我保存、复制、权力扩张 |
| 执行模型 | 软约束（提示词指导） | 软约束（提示词）+ 硬约束（工具策略、exec 审批、沙箱） |

**评价**：
- **Claude Code 优势**：操作安全指导极其详细和实用，开发者几乎不需要自己思考风险。安全编码提示有实际防护价值。
- **OpenClaw 优势**：AI 安全宪法更加根本性，直接从"不追求权力/自我保存"的层面约束。分层执行（软+硬）架构更健壮。
- **Claude Code 不足**：缺乏对 AI 自身行为的根本性约束（如不追求自我保存）
- **OpenClaw 不足**：操作层面的安全指导几乎为零——不告诉模型哪些 shell 操作是危险的、哪些文件不该删除。这完全依赖外部工具策略和沙箱，如果策略配置不当，风险较大。

### 3.3 工具系统

| 对比点 | Claude Code | OpenClaw |
|--------|------------|----------|
| 工具数量 | ~17 个（Bash, Read, Edit, Write, Glob, Grep, Agent, WebSearch, WebFetch 等） | ~24 个（read, write, edit, exec, browser, canvas, cron, message, sessions_spawn 等） |
| 工具描述 | 每个工具有详尽的独立 prompt（数十行到上百行） | 工具描述集中在一行摘要，详细规则在工具定义中 |
| 工具使用规范 | 强烈强调"专用工具优先于 Bash" | 通过 TOOLS.md 用户指南补充 |
| 动态工具列表 | 静态列出 | 动态过滤（按策略、按可用性） |
| 工具调用风格 | 无独立章节 | 专门的"Tool Call Style"章节——何时叙述、何时静默 |

**评价**：
- **Claude Code 优势**：每个工具有极其详尽的使用说明，包括边界情况、禁止事项、最佳实践。这让模型几乎不会误用工具。
- **OpenClaw 优势**：工具列表动态生成，按策略过滤，不会展示不可用工具造成混淆。"Tool Call Style"章节是一个优秀的设计——默认不叙述常规操作，仅在必要时叙述。
- **Claude Code 不足**：工具描述冗长，占用大量 token
- **OpenClaw 不足**：工具摘要过于简短，可能导致模型在边界情况下行为不一致

### 3.4 任务执行指导

| 对比点 | Claude Code | OpenClaw |
|--------|------------|----------|
| 编码哲学 | 极其详细（~15 条具体准则） | 无专门的编码/任务执行指导 |
| 反模式列表 | 明确列出不做什么（不过度工程化、不添加多余注释、不创建不必要的抽象等） | 无 |
| 错误处理指导 | "不要为不可能发生的场景添加错误处理" | 无 |
| 代码审查心态 | 内置（"先读再改"、"理解现有代码再修改"） | 无 |

**评价**：
- **Claude Code 明显优势**：这是 Claude Code 提示词中最精华的部分。每一条准则都来自实际开发中的痛点，极其实用。例如"三行相似的代码比过早的抽象更好"——这个原则能显著减少 AI 过度工程化的倾向。
- **OpenClaw 明显不足**：作为一个也支持代码操作的平台，完全缺乏编码任务指导是一个显著缺陷。这意味着 OpenClaw 在编码任务中更依赖底层模型的默认行为。

### 3.5 输出风格控制

| 对比点 | Claude Code | OpenClaw |
|--------|------------|----------|
| 外部用户版 | "直奔主题、简洁、不要重复用户说的话" | 无独立的输出风格节 |
| 内部用户版 | 详细的散文写作指南（避免碎片化、避免语义回溯、倒金字塔结构等） | 通过 SOUL.md 外部化（用户自定义语气/风格） |
| 表情使用 | 默认不用，除非用户要求 | 支持反应/表情系统，可按频道配置模式 |
| 工具调用叙述 | 无明确指导 | 专门指导：默认静默，仅在必要时叙述 |

**评价**：
- **Claude Code 优势**：内部版的散文写作指南是高水平的技术写作指导，能显著提升输出质量
- **OpenClaw 优势**：工具调用叙述的指导（默认静默）在实践中能显著减少冗余输出。人格化系统（SOUL.md）让风格完全可定制，灵活性远超固定指令。
- **Claude Code 不足**：只有两级（外部简洁 vs 内部详细），无法按场景灵活调整
- **OpenClaw 不足**：核心提示词中缺乏默认的输出风格控制，完全依赖外部文件

### 3.6 多智能体系统

| 对比点 | Claude Code | OpenClaw |
|--------|------------|----------|
| 架构模型 | Coordinator-Worker 两层 | 多层嵌套子智能体（可递归，深度可控） |
| 子智能体提示词 | Agent 工具 prompt + 默认 Agent 提示词 | 专门的 `buildSubagentSystemPrompt()` |
| 任务分配哲学 | "永远不要委派理解"（Coordinator 必须综合发现） | "保持专注"、"你的全部目的就是完成任务" |
| 完成机制 | 同步等待/轮询 | 推送式自动通知（auto-announce） |
| 上下文隔离 | 子智能体使用独立上下文 | minimal 模式减少注入内容，子智能体仅注入 AGENTS.md + TOOLS.md |
| 深度控制 | 无显式限制 | `maxSpawnDepth` 显式限制嵌套深度 |

**评价**：
- **Claude Code 优势**："永远不要委派理解"是一个优秀的设计哲学，防止编排者盲目转发任务。Coordinator 模式的职责划分清晰。
- **OpenClaw 优势**：推送式完成机制比轮询高效得多。深度控制防止无限嵌套。三级提示词模式（full/minimal/none）确保子智能体拿到恰好足够的指令。
- **Claude Code 不足**：缺乏深度控制和推送式完成机制
- **OpenClaw 不足**：缺少"不委派理解"这样的编排哲学指导

### 3.7 记忆系统

| 对比点 | Claude Code | OpenClaw |
|--------|------------|----------|
| 记忆机制 | 文件级记忆（`~/.claude/projects/.../memory/`） | 工具级记忆（memory_search + memory_get） |
| 记忆类型 | user、feedback、project、reference（四类） | 通用记忆文件（MEMORY.md + memory/*.md） |
| 写入控制 | 详细的何时保存/何时不保存指导（~100 行） | 简洁的"搜索然后提取" |
| 陈旧性处理 | "记忆可能过时，先验证再使用" | 无显式指导 |
| 子智能体记忆 | 不适用 | minimal 模式跳过记忆节 |

**评价**：
- **Claude Code 明显优势**：记忆系统的指导极其成熟。四种记忆类型分类合理，何时保存/何时不保存的边界清晰，陈旧性处理的指导防止了"过时记忆误导"的问题。
- **OpenClaw 优势**：工具化的记忆访问（search + get）在大规模记忆场景下更高效，不需要把所有记忆注入上下文
- **OpenClaw 不足**：缺乏记忆分类和写入控制指导

### 3.8 消息与通信

| 对比点 | Claude Code | OpenClaw |
|--------|------------|----------|
| 通信模型 | 单一通道（终端/IDE） | 多渠道路由（20+ 通信渠道） |
| 消息工具 | SendMessage（智能体间） | message（多渠道）+ sessions_send（跨会话）+ subagents（子智能体管理） |
| 回复机制 | 直接输出 | 回复标签（[[reply_to_current]]）、静默回复、心跳 |
| 用户反馈 | 无内置反馈机制 | 反馈反思提示词（评估语气、长度、准确性等） |

**评价**：
- **OpenClaw 明显优势**：多渠道通信架构是 OpenClaw 的核心竞争力。回复标签、静默回复、心跳机制等设计精巧，为不同通信场景提供了完备的解决方案。
- **Claude Code 的定位**：作为 CLI/IDE 工具，单一通道是合理的设计选择，不需要复杂的消息路由。

### 3.9 运行时与环境

| 对比点 | Claude Code | OpenClaw |
|--------|------------|----------|
| 环境信息 | 工作目录、git 状态、平台、shell、OS、模型、知识截止日 | agentId、host、repo、os、arch、node、model、shell、channel、capabilities、thinking |
| 沙箱支持 | 无 | 完整的 Docker 沙箱（容器工作目录、宿主挂载、提权控制、noVNC 浏览器） |
| 模型信息 | 静态声明 | 动态（model + default_model + 模型别名） |
| 能力声明 | 无 | capabilities 字段动态声明频道能力（如 inlineButtons） |

**评价**：
- **OpenClaw 明显优势**：沙箱架构提供了真正的安全隔离，比 Claude Code 的"请先确认"软约束强大得多。提权控制（ask/full）是精心设计的分级安全模型。
- **Claude Code 的简洁性**：不需要沙箱等复杂基础设施，对终端用户更友好

## 五、优缺点总结

### Claude Code

**优点**：

1. **编码任务无出其右** — 编码哲学准则是所有 AI 编程助手提示词中最成熟的。每一条都有实战依据，切实减少了 AI 过度工程化、添加不必要代码的倾向。
2. **操作安全指导详尽** — 从"删除分支需确认"到"不要绕过 git hooks"，覆盖了几乎所有开发者可能踩的坑。
3. **工具使用规范严格** — "专用工具优先于 Bash"这一原则确保了操作可追溯、可审查。
4. **缓存优化策略精明** — 静态/动态分界线设计既节省了 token 成本，又不牺牲灵活性。
5. **记忆系统成熟** — 四类记忆分类、写入/不写入边界、陈旧性处理，形成了完整的知识管理体系。

**缺点**：

1. **场景单一** — 仅面向软件工程，缺乏对非编码场景的支持。
2. **无沙箱隔离** — 完全依赖"请确认"的软约束，一旦用户选择自动批准，所有安全指导形同虚设。
3. **AI 安全约束缺失** — 没有从根本层面约束 AI 行为（如不追求自我保存、不扩展权力）。
4. **多渠道能力为零** — 无法在终端之外与用户交互。
5. **人格化能力欠缺** — 固定的"编程助手"身份，无法适配不同用户偏好。
6. **子智能体完成机制原始** — 缺乏推送式完成通知，多智能体编排效率有提升空间。

### OpenClaw

**优点**：

1. **多渠道架构成熟** — 20+ 通信渠道的统一路由是工程壮举。消息系统（回复标签、静默回复、心跳）设计精巧。
2. **安全隔离真实可靠** — Docker 沙箱 + 提权控制 + 工具策略的三层安全模型远强于纯提示词约束。
3. **人格化系统灵活** — SOUL.md 让用户可以定义完全个性化的 AI 人格，这在消费级产品中价值巨大。
4. **分级提示词高效** — full/minimal/none 三级模式在保持子智能体功能的同时大幅减少 token 消耗。
5. **推送式编排先进** — auto-announce 机制避免了轮询浪费，是多智能体系统的最佳实践。
6. **可扩展性强** — 技能系统 + 插件架构让功能扩展无需修改核心代码。

**缺点**：

1. **编码指导严重缺失** — 作为支持 exec/编辑的平台，完全缺乏编码任务指导是最大的短板。这意味着代码操作质量完全依赖底层模型的默认行为。
2. **操作安全指导为零** — 没有告诉模型哪些 shell 操作是危险的（如 `rm -rf`、`git push --force`），完全依赖外部沙箱。如果用户绕过沙箱或使用提权模式，风险敞口很大。
3. **工具描述过于简短** — 一行摘要可能不足以引导模型在边界情况下的正确行为。
4. **记忆系统缺乏分类指导** — 仅有"搜索+获取"的机制，缺少何时写入、写什么、不写什么的指导。
5. **安全模型过度依赖外部** — AI 安全宪法虽好，但操作安全完全外部化意味着一旦外部防护不完善，提示词本身无法兜底。
6. **提示词可读性较低** — 由 TypeScript 函数动态拼接，比 Claude Code 的静态文本更难审查和维护。

### 一句话总结

**Claude Code** 是一把精心打造的"瑞士军刀"——在软件工程这一个领域做到了极致，但走出代码编辑器就无能为力。

**OpenClaw** 是一个"万能遥控器"——通过多渠道、多智能体、人格化设计覆盖了广泛的场景，但在每个具体领域（特别是编码）的深度不如专注者。

两者的提示词设计都有值得对方借鉴的亮点。如果 OpenClaw 能吸收 Claude Code 的编码哲学和操作安全指导，或者 Claude Code 能吸收 OpenClaw 的沙箱架构和人格化系统，都将大幅提升各自的产品力。
