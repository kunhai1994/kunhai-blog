# Claude Code 系统提示词——架构图与设计深度分析

> 以下通过文本图表 + 文字解读，深度剖析 Claude Code 系统提示词的架构设计思路与亮点。
> 基于 Claude Code v2.1.88 源码逆向还原。

---
**基本思想**
- **是"宪法"** ——绝对不可违反的身份和安全边界，放在最前面确保模型最先处理
  - 你是谁，你的目标，你的任务
  - 你有哪些权利，你能干什么，你不能干什么
- **是"内核"** ——定义了 Agent 如何思考、如何行动的核心逻辑，是整个系统的行为操作系统
  - 你得如何干活，如何与同事（subagent、worker）配合
- **是"UI 层"** ——控制输出的格式和风格，类似 CSS 对 HTML 的作用
  - 你的说话语气、态度
- **是"用户态"** ——每个会话、每个用户都不同的动态信息，运行时注入
  - 如何向用户（老板）汇报、请示

**这种分层确保了：越核心的规则越先被模型"消化"，越外层的信息越容易被动态替换，不影响核心行为。**

```
  缓存层次结构
  ════════════

  ┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐
  │  🌍 Global Cache │     │  🔒 Session Cache │     │  ⚡ Uncached     │
  │                 │     │                  │     │                 │
  │  所有用户共享   │     │  每会话一次计算   │     │  每轮重算       │
  │  仅静态前缀部分 │     │  systemPrompt     │     │  DANGEROUS_     │
  │                 │     │  Section()        │     │  uncached...()  │
  └────────┬────────┘     └────────┬─────────┘     └────────┬────────┘
           │                       │                         │
           │   节省 ~60%           │   MCP 服务器            │
           │   cache_creation      │   连接/断开             │
           │   tokens              │   触发重算              │
           └───────────────────────┴─────────────────────────┘
```

**这个设计牛在哪里**：

1. **三级缓存 = 真金白银**：Anthropic API 的 prompt caching 按 `cache_creation` 收费。静态前缀在所有用户间共享一份缓存 (`cacheScope: 'global'`)，动态部分用 `systemPromptSection()` 做会话级缓存，只有 MCP 指令等极少数内容用 `DANGEROUS_uncachedSystemPromptSection()` 每轮重算。这直接砍掉了大量重复的 token 消耗。
2. **`DANGEROUS_` 前缀是代码级防护**：函数名本身就是警告——调用者必须提供 `reason` 参数说明为什么需要打破缓存，这是一种**让代码审查自动化的命名约定**。
3. **动态边界标记是个精巧的开关**：`shouldUseGlobalCacheScope()` 决定是否插入分割线，在不支持 global cache 的场景下可以静默降级。

## 三、工具路由决策树

```
  用户请求操作
       │
       ├──── 读取文件 ─────────────► [✅ Read 工具]
       │                              支持 PDF/图片/Notebook
       │                              offset+limit 分页
       │
       ├──── 修改文件 ────► 文件是否已读？
       │                       │
       │                    是 ─┤──► [✅ Edit 工具] 精确字符串替换
       │                       │
       │                    否 ─┤──► [❌ 必须先 Read] (errorCode: 6)
       │                       │          │
       │                       │          ▼
       │                       │     [Read] ──► [Edit]
       │
       ├──── 创建新文件 ──────────► [✅ Write 工具]
       │                              要求已有文件先 Read
       │
       ├──── 搜索文件名 ──────────► [✅ Glob 工具]
       │                              glob 模式匹配, 按修改时间排序
       │
       ├──── 搜索文件内容 ────────► [✅ Grep 工具]
       │                              ripgrep 引擎, 正则 + 多行
       │
       ├──── Shell/系统命令 ──────► [⚠️ Bash 工具] 最后手段
       │
       └──── 跨文件/多步骤 ───► 复杂度？
                                   │
                    简单定向搜索 ───┤──► [✅ Glob/Grep 直接搜]
                                   │
                    深度探索 ───────┤──► [🔍 Agent(Explore)]
                                   │      Haiku 模型, 只读工具集
                                   │
                    架构规划 ───────┤──► [📐 Agent(Plan)]
                                   │      只读 + 无 CLAUDE.md
                                   │
                    执行实现 ───────┘──► [🛠️ Agent(general)]
                                          全工具集, 独立子进程
```

**设计亮点**：

1. **"专用工具 > Bash"不是建议，是硬性约束**：提示词用"This is CRITICAL"强调。原因有三——(a) 专用工具调用对用户可见可审查，Bash 是黑盒；(b) 专用工具有结构化输入输出，Bash 是自由文本；(c) 专用工具可以做权限分类（如 Read 自动放行），Bash 需要逐条审核。
2. **Read → Edit 强制依赖链**：通过 `readFileState` Map 实时追踪每个文件的读取时间和内容，Edit 前必须验证。这个设计**从根本上杜绝了 LLM"凭幻觉改代码"的问题**。
3. **三种 Agent 类型形成"知行分离"**：Explore 只能看不能改（用廉价 Haiku 模型），Plan 只做架构不碰代码，General 才有完整执行权。这是经典的**最小权限原则**在 LLM Agent 上的实现。

## 四、安全风控分级模型——可逆性 × 影响范围矩阵

```
  不可逆 ▲
        │
        │  rm -rf            DROP TABLE
        │    ●                  ●
        │
        │  git reset    git push --force     删除远程分支
        │  --hard ●          ●                    ●
        │                                    修改 CI/CD
        │                                        ●
        │                 git push      发送 Slack 消息
        │                    ●               ●
        │
        │  git commit       创建 PR
        │    ●                ●
        │
        │
        │  编辑本地文件
        │    ●
        │  运行测试
        │    ●
        │  读取文件
        │    ●  Glob/Grep
        │       ●
   可逆 │
        └──────────────────────────────────────────────►
              本地影响                            全局影响

        ┌──────────────────┬──────────────────────┐
        │  🟢 自动放行      │  🟡 需要确认          │
        │  (本地+可逆)      │  (全局+可逆)          │
        ├──────────────────┼──────────────────────┤
        │  🟡 默认确认      │  🔴 必须确认 + 警告   │
        │  (本地+不可逆)    │  (全局+不可逆)        │
        └──────────────────┴──────────────────────┘
```

```
  风控决策流程
  ════════════

  Agent 即将执行操作
         │
         ├── 操作可逆？
         │      │
         │   是 ─┤──── 影响范围？
         │      │         │
         │      │      仅本地 ──────► [🟢 自动执行, 无需确认]
         │      │         │
         │      │      影响他人 ────► [🟡 询问确认, 透明告知操作]
         │      │      (push/PR/消息)
         │      │
         │   否 ─┤──── 用户明确要求？
         │               │
         │            否 ─┤──► [🔴 拒绝执行, 建议安全替代方案]
         │               │
         │            是 ─┤──── 是否 force push main？
         │                        │
         │                     是 ─┤──► [⚠️ 强制警告, 即使用户要求也提醒]
         │                        │
         │                     否 ─┘──► [🟡 执行但记录原因]
```

**设计思路解读**：

这个风控模型的核心思想是**"暂停确认的代价很低，但错误操作的代价可能极高"**（原文："The cost of pausing to confirm is low, while the cost of an unwanted action can be very high"）。

亮点在于几个反直觉的设计：

1. **"一次授权 ≠ 永久授权"**：用户批准过一次 `git push` 不代表以后所有 push 都自动通过。这打破了传统 sudo 式的"信任升级"模式。
2. **"发现异常 → 调查而非删除"**：遇到陌生文件/分支/锁文件时，提示词要求先调查再行动。这等于给 LLM 植入了一个"好奇心优先于清理欲"的人格特质。
3. **Bash 工具的 29 项安全检测**：包括命令替换模式 `$()`、Zsh 危险命令 `zmodload`、IFS 注入、Unicode 空白字符混淆等。这是**生产级的 shell 注入防护**。

**为什么这个设计很强**：

1. **"永远不要委派理解"**（Never delegate understanding）是整个多 Agent 系统最精髓的一句话。大多数 Agent 框架的反模式就是 `"根据你的发现，去修复它"`——把综合和决策都推给了子 Agent。Claude Code 要求 Coordinator 必须**先综合、再精确指派**，包含文件路径、行号、具体改什么。
2. **"并行是超能力"**：Coordinator 被明确告知要同时启动独立的 Worker，而不是串行等待。这在 token 上更贵，但在**墙钟时间**上大幅更快。
3. **Coordinator 自己几乎没有执行工具**：只有 Agent、SendMessage、TaskStop 三件套。这是**纯粹的指挥官模式**——思考、综合、指挥，不亲自下场。

```
     │  ┌──────────────┐    │
     │  │ 理解 → 实现  │    │
     │  └──────┬───────┘    │
     │         ▼            │
     │  ┌──────────────┐    │          ┌─────────────────────────┐
     │  │ 运行测试     ├────┤─ 到达 ──►│ 检查 terminalFocus      │
     │  └──────┬───────┘    │  自然    │                         │
     │         │ 通过       │  节点    │  unfocused (用户走了):   │
     │         ▼            │          │    → 自主决策            │
     │  ┌──────────────┐    │          │    → 直接 commit/push   │
     │  │ 提交         │    │          │    → 只在不可逆操作暂停  │
     │  └──────────────┘    │          │                         │
     └──────────────────────┘          │  focused (用户在看):     │
                                       │    → 呈现选项           │
                                       │    → 大变更前先确认      │
                                       │    → 保持简洁输出       │
                                       └─────────────────────────┘

     Sleep 的成本博弈:
     ┌─────────────────────────────────────────────────────┐
     │  Sleep 太短  →  频繁唤醒  →  每次 = 1次API调用 = 💰 │
     │  Sleep 太长  →  cache 过期 (>5min) → 重建缓存 = 💰💰│
     │  输出废话    →  "仍在等待" = 浪费 output token = 💰  │
     │                                                     │
     │  最佳策略: 有活干 → 短 Sleep (30-60s)               │
     │           等慢操作 → 中 Sleep (2-3min)               │
     │           彻底没事 → 接近 5min 上限                  │
     └─────────────────────────────────────────────────────┘
```

**这里的设计智慧**：

1. **终端焦点感知**是一个极其细腻的 UX 设计——当用户的终端窗口失焦（切到浏览器/聊天工具了），Agent 切换为高度自主模式，大胆决策、提交、推送；当用户盯着终端时，变得更协作、更透明。这种**根据用户注意力调节自主度**的机制，我在其他 AI Agent 产品中没见过。
2. **"如果无事可做，必须调用 Sleep"**——这不是废话，这是在省钱。每次 Agent 回复都是一次 API 调用，输出"仍在等待"就是烧 token 打水漂。而 Sleep 工具让 Agent 可以指定等待时长，平衡"响应速度"和"API 成本"。
3. **prompt cache 5 分钟过期**这个数字被写进提示词，是因为 Agent 需要根据这个数字决定 Sleep 多久——Sleep 太久缓存过期了，下次唤醒要重新创建缓存（更贵）；Sleep 太短唤醒太频繁（也更贵）。这种**让 LLM 参与成本优化决策**的做法非常先进。

**设计亮点**：

1. **安全工具的白名单短路**：Read、Grep、Glob 等只读工具直接放行，不进分类器。这不仅是性能优化（省去分类延迟），更是 UX 优化——用户不会被"是否允许读取文件？"这种低风险确认打断。
2. **"被拒绝后不重试，而是反思"**：这个规则把 LLM 从"执行者"提升为"思考者"——它必须理解用户为什么拒绝，然后调整方案。这比 `while (!success) retry()` 的机械重试高级太多。
3. **Bash 工具的 29 项安全检测**本质上是在 LLM 和 Shell 之间加了一层 **WAF (Web Application Firewall)**。涵盖了命令注入的几乎所有已知攻击面，包括一些非常冷门的攻击手法如 Unicode 空白字符混淆和中间 hash 注释注入。

**为什么需要延迟加载**：

**这是 Claude Code 最精妙的安全设计之一**：

LLM 的核心弱点之一是**幻觉**——它可能"记得"一个函数的签名，但实际代码早已改变。强制 Read-before-Edit 机制通过 `readFileState` Map 做了三重保护：

1. **时间戳验证**：文件在 Read 和 Edit 之间如果被外部修改（用户手动改的、其他 Worker 改的），Edit 会拒绝执行
2. **部分读取检测**：如果用了 `offset/limit` 只读了文件的一部分，`isPartialView=true` 也会阻止 Edit（因为你没看到全貌）
3. **内容校验兜底**：在 Windows 等时间戳不可靠的系统上，还会做 content 比较作为 fallback

#### 十大核心设计亮点

| # | 亮点 | 解读 |
|---|------|------|
| **1** | **三级缓存分割** | 静态前缀全局共享、会话缓存、每轮重算三层，用 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 分割，直接降低 API 的 `cache_creation` 成本 |
| **2** | **Read-before-Edit 强制链** | 通过 `readFileState` Map 追踪每个文件的读取状态，从根本上防止 LLM 基于幻觉修改代码 |
| **3** | **"永远不要委派理解"** | Coordinator 必须先综合 Worker 的发现，用精确的文件路径+行号+具体改动指派下一步，杜绝"根据你的发现去修"这种甩锅 |
| **4** | **终端焦点自适应** | 根据用户是否在看终端，动态切换"高度自主"和"协作确认"模式——AI Agent 领域首创的注意力感知 UX |
| **5** | **Bash 29 项安全检测** | 涵盖命令替换、Zsh 特殊命令、IFS 注入、Unicode 混淆等几乎所有 Shell 注入攻击面，等同于 LLM 和 Shell 之间的 WAF |
| **6** | **可逆性×爆炸半径矩阵** | 不是简单的"安全/危险"二分法，而是按操作的可逆程度和影响范围做二维评估，更精确地控制确认策略 |
| **7** | **"不做过度工程"写进宪法** | 明确写了"三行重复好过过早抽象"、"不为假想未来设计"、"不加不可能的错误处理"——这是给 LLM 的反过度工程疫苗 |
| **8** | **工具延迟加载 + 预算控制** | 延迟加载防止 MCP 工具撑爆上下文，Skill 列表限制在上下文窗口的 1%，精确控制提示词膨胀 |
| **9** | **内外用户双版本输出策略** | 外部用户版强调"直奔主题"，内部 Anthropic 用户版强调"流畅散文 + 为冷启动读者写作"——同一系统两套表达风格 |
| **10** | **Sleep 工具的成本意识** | 把 prompt cache 5 分钟过期这个基础设施参数暴露给 LLM，让模型自己决定等多久——LLM 首次被纳入成本优化回路 |

**关键的架构决策**：

- **两条完全独立的路径**：自主模式 (KAIROS) 和标准模式的提示词组装是两套不同的逻辑，而非在标准路径上打补丁。这避免了条件分支在复杂场景下的组合爆炸。
- **`resolveSystemPromptSections()` 是最终网关**：无论哪条路径，最终都经过这个函数过滤空值和展平数组，确保输出干净。
- **MCP 指令被标记为 `DANGEROUS_uncached`**：因为 MCP 服务器可能随时连接/断开，必须每轮重算。函数名中的 `DANGEROUS_` 前缀强制开发者在代码审查时关注这个成本。
