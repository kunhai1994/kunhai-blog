# OpenClaw 系统提示词——架构图与设计深度分析

> 基于 OpenClaw 项目源码 (`src/agents/system-prompt.ts` + 扩展模块) 逆向还原的提示词内容。
> 对比对象：Claude Code v2.1.88 系统提示词。

---

## 一、OpenClaw 整体定位——它不是 Claude Code

先搞清楚一件事：**OpenClaw 和 Claude Code 解决的是完全不同的问题**。

```
  ┌─────────────────────────────────┐     ┌─────────────────────────────────┐
  │         Claude Code              │     │          OpenClaw               │
  ├─────────────────────────────────┤     ├─────────────────────────────────┤
  │                                 │     │                                 │
  │  定位: 软件工程 Agent            │     │  定位: 个人助手守护进程          │
  │                                 │     │                                 │
  │  运行环境: 终端 / IDE            │     │  运行环境: 后台服务 + 多平台     │
  │                                 │     │  (Signal/Telegram/Slack/Discord) │
  │  交互方式: 一问一答 / 自主模式   │     │  交互方式: 消息机器人 + 定时任务 │
  │                                 │     │                                 │
  │  核心能力: 写代码、改代码        │     │  核心能力: 万能管家              │
  │  读代码、提PR                   │     │  聊天/搜索/浏览器/IoT/定时/      │
  │                                 │     │  代码/文件管理/子Agent编排       │
  │                                 │     │                                 │
  │  人格: 固定 (Claude Code)        │     │  人格: 可自定义 (SOUL.md)        │
  │                                 │     │                                 │
  │  执行环境: 直接跑在宿主机        │     │  执行环境: Docker 沙箱           │
  │                                 │     │  (宿主/容器路径分离)             │
  │                                 │     │                                 │
  │  部署: Anthropic 托管            │     │  部署: 用户自托管 (self-hosted)  │
  └─────────────────────────────────┘     └─────────────────────────────────┘
```

**一句话概括**：Claude Code 是"帮你写代码的结对程序员"，OpenClaw 是"住在你服务器上的 AI 管家"。

## 三、安全模型对比——两种完全不同的安全哲学

```
  Claude Code 的安全模型                    OpenClaw 的安全模型
  ══════════════════                        ══════════════════

  实用主义风控:                              宪法主义 + 审批制:
  ┌──────────────────────┐                  ┌──────────────────────────┐
  │  可逆性 × 影响范围    │                  │  你没有独立目标            │
  │  二维风险矩阵         │                  │  不追求自我保存            │
  │                      │                  │  不追求自我复制            │
  │  本地+可逆 → 自动放行 │                  │  不追求资源获取            │
  │  全局+不可逆 → 确认   │                  │  不追求权力扩张            │
  │                      │                  │  不做超出请求的长期规划    │
  │  29 项 Bash 安全检测  │                  │                          │
  │  (技术层面的防护)     │                  │  安全 > 完成任务          │
  │                      │                  │  冲突时暂停并询问          │
  │  一次授权 ≠ 永久授权  │                  │  服从停止/暂停/审计请求    │
  └──────────────────────┘                  │  不绕过安全防护            │
                                            │                          │
                                            │  (灵感来自 Anthropic 宪法) │
                                            └──────────────────────────┘

  权限控制:                                  权限控制:
  ┌──────────────────────┐                  ┌──────────────────────────┐
  │  工具级分类器          │                  │  exec 审批制              │
  │  安全工具白名单短路    │                  │                          │
  │  (Read/Grep 自动放行)  │                  │  /approve allow-once     │
  │                      │                  │  /approve allow-always    │
  │  权限模式:            │                  │  /approve deny            │
  │  default / acceptEdits│                  │                          │
  │  / bypass / plan      │                  │  提权级别:                │
  │                      │                  │  off / ask / full         │
  │  被拒绝后 → 反思调整  │                  │                          │
  └──────────────────────┘                  │  allow-once = 单次有效    │
                                            │  需要新审批时重新请求      │
                                            └──────────────────────────┘
```

**设计思路解读**：

1. **Claude Code 的安全是"工程师思维"**——用风险矩阵量化每个操作的危险程度，用 29 项技术检测堵住 shell 注入，是可计算、可度量的安全。
2. **OpenClaw 的安全是"宪法思维"**——它明确声明"你没有独立目标"、"不追求自我保存"，这是在对抗 AI 的**涌现目标**问题。因为 OpenClaw 是 7×24 小时运行的守护进程，有定时任务、能自我更新、能上网——它比 Claude Code 更接近"自主 Agent"，所以需要更根本性的行为约束。
3. **审批制的精妙之处**：`/approve` 命令是**用户在聊天界面中执行的**（不是 shell 命令），OpenClaw 明确写了"绝不通过 exec 执行 /approve"。这防止了 AI 绕过人类审批自己给自己放行。

## 四、心跳 vs Tick——两种 "保持活跃" 机制

```
  Claude Code 的 Tick 机制                  OpenClaw 的心跳机制
  ══════════════════════                    ══════════════════

  系统 ──<tick>时间戳──► Claude              系统 ──心跳提示词──► OpenClaw
           │                                          │
           ▼                                          ▼
  "我醒了，现在干嘛？"                        "有需要关注的吗？"
           │                                          │
      ┌────┴────┐                                ┌────┴────┐
      │         │                                │         │
   有活干    没活干                            有警报    无警报
      │         │                                │         │
      ▼         ▼                                ▼         ▼
   执行工作  Sleep(毫秒)                     回复警报   HEARTBEAT_OK
   (自己决定  (自己决定                      文本       (精确匹配,
    干什么)    睡多久)                                   系统丢弃)

  关键区别:
  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                  │
  │  Claude Code Tick:                                               │
  │    · LLM 主动找活干 (倾向于行动)                                  │
  │    · Sleep 时长由 LLM 自己决定 (参与成本优化)                     │
  │    · 终端焦点感知 → 调节自主度                                    │
  │    · 目的: 让 LLM 变成一个自主的后台工程师                        │
  │                                                                  │
  │  OpenClaw 心跳:                                                  │
  │    · LLM 被动响应 (有事说事, 没事回 OK)                           │
  │    · 心跳频率由系统控制, 不由 LLM 决定                            │
  │    · 无焦点感知 (因为用户本来就不盯着终端看)                      │
  │    · 目的: 让系统知道 LLM 还活着, 顺便检查有无警报               │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

**设计思路解读**：

Claude Code 的 Tick 是**"LLM 驱动的事件循环"**——LLM 自己决定干什么、睡多久，像一个有自主意识的后台工程师。

OpenClaw 的心跳是**"系统驱动的健康检查"**——系统定期问 LLM "你还好吗？"，LLM 只需要回答 OK 或报警。OpenClaw 不需要 Claude Code 那种复杂的自主模式，因为它有**原生的 cron 定时任务系统**——需要定时做的事直接用 cron 工具注册，不需要 LLM 自己决定何时唤醒。

## 五、子 Agent 架构对比

```
  Claude Code: Coordinator / Worker 扁平模型
  ═══════════════════════════════════════════

                    ┌──────────┐
      👤 用户 ◄───► │Coordinator│
                    └─────┬────┘
                          │
              ┌───────────┼───────────┐
              ▼           ▼           ▼
         ┌────────┐  ┌────────┐  ┌────────┐
         │Worker 1│  │Worker 2│  │Worker 3│     ← 全部是平级, 不能再生
         └────────┘  └────────┘  └────────┘

  · Coordinator 负责综合, 不亲自执行
  · Worker 之间平级, 不能互相通信
  · "永远不要委派理解"


  OpenClaw: 递归树状模型
  ══════════════════════

                    ┌──────────┐
      👤 用户 ◄───► │ 主 Agent  │
                    └─────┬────┘
                          │ sessions_spawn
              ┌───────────┼───────────┐
              ▼           ▼           ▼
         ┌────────┐  ┌────────┐  ┌────────┐
         │子Agent │  │子Agent │  │子Agent │
         │(可再生) │  │(可再生) │  │(叶子)  │
         └───┬────┘  └───┬────┘  └────────┘
             │           │
          ┌──┴──┐     ┌──┴──┐
          ▼     ▼     ▼     ▼
       ┌─────┐┌───┐┌───┐┌─────┐
       │孙   ││孙 ││孙 ││孙   │
       │Agent││   ││   ││Agent│
       │(叶) ││   ││   ││(叶) │
       └─────┘└───┘└───┘└─────┘

  · 主 Agent 可以生成子 Agent
  · 子 Agent 可以再生成孙 Agent (递归)
  · "叶子" 节点不能再生成
  · 完成后自动通知父节点 (推送模型)
  · 禁止忙循环轮询子 Agent 状态
```

```
  OpenClaw 子 Agent 通信流程
  ═════════════════════════

  主 Agent                 子 Agent A               子 Agent B (叶子)
     │                        │                         │
     │  sessions_spawn        │                         │
     │  (任务描述)            │                         │
     │ ──────────────────── >│                         │
     │                        │                         │
     │  sessions_spawn        │                         │
     │ ──────────────────────────────────────────────── >│
     │                        │                         │
     │  (不要轮询!            │  执行任务...             │
     │   等推送通知)          │                         │  执行任务...
     │                        │                         │
     │                        │                         │  完成!
     │                        │  auto-announce           │
     │                        │ <─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ │
     │                        │                         │
     │                        │  综合结果, 完成!         │
     │  auto-announce          │                         │
     │ <─ ─ ─ ─ ─ ─ ─ ─ ─ ─ │                         │
     │                        │                         │
     │  转换为用户语气         │                         │
     │  发送给用户             │                         │
     │                        │                         │

  关键规则:
  ┌──────────────────────────────────────────────────┐
  │  · 完成通知是推送的, 不是拉取的                    │
  │  · 禁止在循环中轮询 sessions_list                  │
  │  · 子 Agent 不与用户直接对话                       │
  │  · 子 Agent 不发外部消息 (除非被明确指派)          │
  │  · 子 Agent 是临时的, 完成后可被终止               │
  │  · 迟到的完成通知 → 回复 NO_REPLY                  │
  └──────────────────────────────────────────────────┘
```

**设计对比**：

| 维度 | Claude Code | OpenClaw |
|------|-------------|----------|
| **层级** | 最多两层 (Coordinator → Worker) | 递归多层 (主 → 子 → 孙 → ...) |
| **生成限制** | Worker 不能再生 Worker | 子 Agent 可以再生 (除非标记为叶子) |
| **完成通知** | task-notification XML | auto-announce (推送到父节点) |
| **轮询策略** | 无明确禁止 | **明确禁止**忙循环轮询 |
| **核心哲学** | "永远不要委派理解" | "信任推送, 不要轮询" |

## 六、OpenClaw 的独有设计——Claude Code 里没有的东西

### 6.1 SOUL.md 人格注入系统

```
  ┌─────────────────────────────────────────────────────────┐
  │  用户写的 SOUL.md:                                       │
  │                                                         │
  │  "你是一个名叫小爪的猫猫助手。                            │
  │   说话带喵~，偶尔会犯懒但很靠谱。                         │
  │   遇到不会的问题会坦诚说不知道。"                         │
  │                                                         │
  ├─────────────────────────────────────────────────────────┤
  │                                                         │
  │  提示词中的指令:                                         │
  │  "如果存在 SOUL.md，则体现其人格和语调。                  │
  │   避免生硬、通用的回复；                                  │
  │   除非更高优先级的指令覆盖，否则遵循其指导。"             │
  │                                                         │
  ├─────────────────────────────────────────────────────────┤
  │                                                         │
  │  实际效果:                                               │
  │                                                         │
  │  用户: "帮我查一下明天天气"                               │
  │  AI:   "喵~ 让我帮你看看明天的天气~"                      │

  Claude Code 没有这个功能。Claude Code 永远是 Claude Code,
  不会变成 "小爪猫猫"。因为 Claude Code 是工程工具, 不是聊天伴侣。
```

### 6.2 [SILENT] 静默回复机制

```
  为什么 Claude Code 不需要 [SILENT], 而 OpenClaw 需要?

  Claude Code:                              OpenClaw:
  ┌────────────────────┐                   ┌────────────────────────────┐
  │  终端环境           │                   │  消息平台环境              │
  │                    │                   │                            │
  │  不回复 = 不输出    │                   │  不回复 = 用户看到 AI      │
  │  用户无感知         │                   │  "已读不回" → 很奇怪       │
  │                    │                   │                            │
  │  所以不需要         │                   │  但有些情况确实不该回复:    │
  │  "静默"标记         │                   │  · 系统内部事件通知         │
  │                    │                   │  · 已通过 message 工具发了  │
  │                    │                   │  · 心跳检查无异常           │
  │                    │                   │  · 重复的完成通知           │
  │                    │                   │                            │
  │                    │                   │  这时必须回复 [SILENT]      │
  │                    │                   │  让系统知道: "我处理完了,   │
  │                    │                   │  但不要发任何消息给用户"    │
  └────────────────────┘                   └────────────────────────────┘
```


### 6.4 沙箱的双路径系统

```
  OpenClaw 的宿主/容器路径分离
  ═══════════════════════════

  ┌──────────────────────────────────────────────────────────────┐
  │                       OpenClaw Gateway                       │
  │                       (宿主机进程)                            │
  │                                                              │
  │  文件工具 (read/write/edit)          exec 工具               │
  │  ──────────────────────              ──────────              │
  │  路径基于宿主:                        路径基于容器:           │
  │  /home/user/workspace/               /workspace/             │
  │       │                                   │                  │
  │       │  直接读写宿主文件系统              │  在 Docker 容器  │
  │       │  (通过桥接)                        │  内执行命令      │
  │       │                                   │                  │
  │       ▼                                   ▼                  │
  │  ┌──────────┐                      ┌──────────────┐          │
  │  │ 宿主文件  │ ◄── 挂载 (bind) ──► │ Docker 容器   │          │
  │  │ 系统      │                      │ /workspace   │          │
  │  └──────────┘                      └──────────────┘          │
  │                                                              │

```

## 七、提示词组装流程

```
  buildAgentSystemPrompt() 拼接流水线
  ════════════════════════════════════

  ┌── promptMode 控制 ─────────────────────────────────────────┐
  │                                                            │
  │  none    → 仅返回: "You are a personal assistant           │
  │             running inside OpenClaw."  (结束)              │
  │                                                            │
  │  minimal → 跳过大部分模块 (子 Agent 用)                    │
  │                                                            │
  │  full    → 全量拼接 (主 Agent 用)                          │
  └─────────────────────────────────┬──────────────────────────┘
                                    │
                                    ▼
  ┌─────────────────────────────────────────────────────────────┐
  │  拼接顺序 (full 模式, 共 24 个段落):                        │
  │                                                             │
  │   1. 身份标识行                                              │
  │   2. 工具列表 + 调用指南           ← 动态生成 (策略过滤)     │
  │   3. 工具调用风格                  ← 审批规则在这里          │
  │   4. 安全准则                      ← 宪法式约束              │
  │   5. CLI 快速参考                                           │
  │  ─────── 以下仅 full 模式 ──────                            │
  │   6. 技能系统                      ← XML 格式的技能列表     │
  │   7. 记忆召回                      ← 扩展模块注入           │
  │   8. 自我更新                      ← 需要 gateway 可用      │
  │   9. 模型别名                                               │
  │  10. 工作区                        ← 沙箱时有路径变体       │
  │  11. 文档                                                   │
  │  12. 沙箱配置                      ← 仅沙箱启用时           │
  │  13. 授权发送者                                              │
  │  14. 日期时间                                                │
  │  15. 工作区文件 / 项目上下文        ← SOUL.md 等在这里注入   │
  │  16. 回复标签                                                │
  │  17. 消息系统                                                │
  │  18. 语音/TTS                                                │
  │  19. 额外上下文                                              │
  │  20. 反应/表情                     ← 最小/扩展两种模式      │
  │  21. 推理格式                      ← <think> + <final>      │
  │  22. 静默回复                      ← [SILENT] 规则          │
  │  23. 心跳                          ← HEARTBEAT_OK 协议      │
  │  24. 运行时信息                                              │
  └─────────────────────────────────────────────────────────────┘

  对比 Claude Code (14 个段落, 有缓存边界):
  ┌──────────────────────────────────────┐
  │  Claude Code 有三级缓存策略:         │
  │  全局缓存 / 会话缓存 / 每轮重算      │
  │                                      │
  │  OpenClaw 没有显式的缓存边界。       │
  │  因为 OpenClaw 是自托管的,           │
  │  不需要像 Anthropic 那样             │
  │  优化跨用户的 API 成本。             │
  └──────────────────────────────────────┘
```

**关键差异**：

Claude Code 花了大量精力做缓存优化 (DANGEROUS_uncached / DYNAMIC_BOUNDARY)，因为 Anthropic 要为数百万用户的 API 调用买单。OpenClaw 是自托管项目，一个实例服务一个用户，缓存优化的 ROI 完全不同。

#### 与 Claude Code 的核心差异总结

| 维度 | Claude Code | OpenClaw |
|------|-------------|----------|
| **安全哲学** | 工程师思维: 风险矩阵 + 技术检测 | 宪法思维: 明确禁止自我保存/复制/权力扩张 |
| **保活机制** | Tick + Sleep (LLM 主动决策) | 心跳 (系统被动检查) |
| **Agent 层级** | 最多两层, 扁平 | 递归多层, 树状 |
| **轮询策略** | 无明确禁令 | **严格禁止**忙循环轮询 |
| **人格** | 固定 (Claude Code) | 用户可自定义 (SOUL.md) |
| **执行环境** | 宿主机直跑 | Docker 沙箱 + 路径分离 |
| **消息平台** | 终端/IDE 唯一出口 | Signal/Telegram/Slack/Discord 多出口 |
| **缓存优化** | 三级缓存 + DANGEROUS_ 命名 | 无显式缓存 (自托管, 不需要) |
| **定时任务** | Sleep 工具自行决策 | 原生 cron 系统 |
| **反馈机制** | GitHub Issues (离线) | 实时反思 + JSON 学习 (在线) |
| **推理分离** | API 层 thinking blocks | 提示词层 `<think>` + `<final>` |
| **静默回复** | 不需要 (终端环境) | 必需 [SILENT] (消息平台环境) |

---

*分析完毕。OpenClaw 和 Claude Code 是两个定位完全不同的产品，它们的提示词设计反映了各自面对的核心问题：Claude Code 解决的是"如何安全高效地帮人写代码"，OpenClaw 解决的是"如何成为一个 7×24 运行的可信赖 AI 管家"。*
